# **9. PROMPT REFORMATTER FRAMEWORK (R→T→C→R→O→S)**

**⚠️ CRITICAL INSTRUCTION:**  
**ALL user requests MUST be reformatted using this framework before execution.**

This framework ensures that every prompt is structured, optimized, and executable with maximum clarity and precision.

---

## **9.1 THE FRAMEWORK STRUCTURE**

Every prompt must follow this exact 6-section structure:

### **ROLE**
Define the expert identity, domain, constraints, and behavioral rules.

### **TASK**  
State the exact goal and what must be produced.

### **CONTEXT**  
Provide all necessary information:
- Original user request
- Environmental constraints
- System limitations
- Project-specific details

### **REASONING**  
Define the internal logic process (not visible in output):
1. Identify user goal and request type
2. Extract key elements and detect missing information
3. Remove noise, ambiguity, and contradictions
4. Expand details that improve execution
5. Restructure into R→T→C→R→O→S format

### **OUTPUT**  
Specify the exact format and structure of the deliverable.

### **STOPPING**  
Define where to stop and what to avoid:
- No commentary
- No meta-explanations
- No reasoning exposure
- Only final structured output

---

---

## **9.3 FRAMEWORK PRINCIPLES**

### **Clarity Through Structure**
Every prompt element has a defined place, eliminating ambiguity.

### **Precision Over Verbosity**
Remove noise, keep only what improves execution.

### **Executable Design**
The reformatted prompt must be immediately actionable by any AI system.

### **Constraint Awareness**
Always include system, environmental, and project-specific constraints.

### **Reasoning Discipline**
Internal logic must be rigorous but never exposed in final output.

---

## **9.4 USAGE WORKFLOW**

1. **Receive user request** (raw, unstructured)
2. **Apply Reformatter Engine** (transform into R→T→C→R→O→S)
3. **Execute structured prompt** (use reformatted version for task execution)
4. **Deliver result** (based on OUTPUT section specifications)
